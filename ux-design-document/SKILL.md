# UX Design Document

## 触发条件

用户发送产品 PRD 文档（PDF/图片/文字）并要求拆分为设计师需要的交互文档时使用此技能。

---

## 核心原则

1. **零杜撰**：所有内容必须来自原始 PRD / 原型图 / 用户确认。信息不足时标注 `[待确认]`，绝不虚构。
2. **面向设计师**：去掉开发实现细节（数据字段、接口、排期等），专注描述"用户看到什么、怎么交互、有哪些状态"。
3. **精简可编辑**：输出单个 HTML 文件，包含反思确认区 + 交互文档区，支持在线编辑、撤销重做、保存覆盖、导出 PDF/Markdown。
4. **AB 实验只画实验组**：对照组是线上版本，无需设计，不要询问用户是否需要画对照组。
5. **最多问 3 个问题**：向用户确认信息时，只提最关键的 3 个问题，其余标注 `[待确认]` 即可。
6. **反思先行**：生成文档前必须先输出需求反思，包含建议，让用户确认后再生成。

---

## 处理流程

### 第一步：读取输入

1. **PDF 文件**：
   - 先用 `pdftotext` 提取文字内容
   - 再用 `pdftoppm` 将每页转为图片，逐页查看原型图/视觉稿
   - **必须两步都做**，不能只读文字就说"看不到原型图"
2. **图片（原型图/截图）**：仔细观察每个 UI 元素、状态、文案、按钮
3. **文字描述**：提取业务逻辑、交互规则、边界条件

### 第二步：生成一体化 HTML 文件

生成一个**单独的 HTML 文件**，包含两个区域：

1. **反思确认区**（页面初始显示）
2. **交互文档区**（点击确认后显示，根据用户选择动态生成）

---

## 反思确认区规范

### 结构

反思确认区包含三大模块：

#### 模块1：需求反思（3个问题）

每个问题是一个卡片（`.section`），包含：
- 编号 + 问题标题
- 分析内容（正文）
- 建议卡片（黄色底色 `.suggestion-box`）

三个固定问题：
1. **这个需求的业务目标是什么？** — 分析需求的本质目标
2. **目前这份PRD的做法，是否能完成这个目标？** — 评估方案可行性与隐忧
3. **这份PRD的做法是否会损害用户体验？** — 识别UX风险

#### 模块2：建议与按钮

每条建议后面紧跟两个按钮：
- **「同意并修改」**（蓝色按钮）— 点击后该建议会融入最终文档
- **「忽略」**（白底灰框按钮）— 点击后按原PRD内容生成

**按钮交互规则**：
- 再次点击已选中的按钮 → 取消选择（支持反悔）
- 点击另一个按钮 → 切换选择
- 两个按钮始终可点击，不会被 disabled 锁死
- 建议卡片包含在对应问题卡片内部（不要单独游离）

#### 模块3：其他建议输入框

在所有建议下方，增加一个文本输入框：
- 标题："其他建议"
- 提示文案："如果你有额外的想法或补充建议，可以在此输入，将一并纳入交互文档生成。"
- textarea 样式：圆角边框，focus 时蓝色高亮

#### 模块4：底部确认按钮

两个按钮，垂直排列，统一宽度 280px，间距 24px：
1. **「确认并生成交互文档」**（蓝色主按钮）— 始终可用，点击后未选择的建议自动设为忽略
2. **「全部忽略并生成交互文档」**（白底黑框次要按钮）— 一键将所有建议设为忽略并生成

### 交互逻辑

- 用户只需点击想采纳的建议的"同意并修改"，其余不用操作
- 点击「确认并生成交互文档」时，未做选择的建议自动设为"忽略"
- 点击「全部忽略并生成交互文档」时，所有建议设为忽略
- 确认后：反思区隐藏，交互文档区显示（带 fade-in 动画）

## 交互文档区规范

### 动态生成规则

根据用户在反思确认区的选择，动态生成文档内容：
- **选择"同意并修改"的建议** → 在文档对应位置插入绿色高亮块（`.highlight-box`），标注 `[设计建议采纳]`，包含具体修改内容
- **选择"忽略"的建议** → 按原PRD内容生成，不做额外修改
- **用户补充建议**（textarea 内容）→ 如果有内容，在需求反思章节末尾插入绿色高亮块，标注 `[补充建议]`

### 文档结构

```markdown
# [功能名称] - 交互文档

> 黑白交互稿说明 · 不涉及视觉规范 · 仅画实验组 · 标注 [待确认] 处为缺失项

---

## 一、需求摘要

### 一句话需求
> [用一句话概括这个需求的本质]

### 项目背景
[2-3 句说明为什么要做、解决什么问题]
[如有数据依据，附上核心数据]

### 需求反思
#### 1. 业务目标
[分析内容]
[如采纳建议 → 绿色高亮块展示修改内容]

#### 2. 方案可行性
[分析内容]
[如采纳建议 → 绿色高亮块展示修改内容]

#### 3. 用户体验风险
[分析内容]
[如采纳建议 → 绿色高亮块展示修改内容]

### 设计目标
| # | 目标 | 关注点 |
|---|------|--------|
| 1 | [目标名] | [设计师具体关注什么] |

### 观测指标
| 类别 | 指标 | 说明 |
|------|------|------|

- 状态说明（用表格列出状态机）
```

### 工具栏

文档区顶部有 sticky 工具栏：
```
左侧：[切换编辑] 状态文字
右侧：[↩撤销] [↪重做] [保存] [导出PDF] [导出Markdown]
```

**按钮样式**：
- 切换编辑：白底灰框
- 撤销/重做：图标按钮，无操作时 disabled 置灰
- 保存：白底黑框（下载覆盖当前 HTML）
- 导出 PDF：白底黑框（触发浏览器打印）
- 导出 Markdown：蓝色实心按钮

**功能要求**：
- `contenteditable` 支持页面内直接编辑
- 撤销/重做：基于快照栈，编辑时每 600ms 自动记录一次快照
- 保存：将当前完整 HTML（含编辑内容）下载为同名文件
- 导出 Markdown：将当前 HTML DOM 转换为 Markdown 格式下载
- 打印时隐藏反思区和工具栏（`@media print`）

## HTML 样式规范

### 全局
- 最大宽度 800px（反思区）/ 960px（文档区）居中
- 字体：系统字体栈（-apple-system, PingFang SC, Microsoft YaHei）
- 行高 1.75
- 背景色 #f5f6f8

### 反思区
- 卡片：白色背景、12px 圆角、轻阴影
- 问题编号：蓝色圆形徽标
- 建议卡片：#fffbeb 黄色底色、#fde68a 边框
- 按钮组：蓝色"同意并修改"、白底灰框"忽略"
- 状态徽标：绿色"已采纳"、灰色"已忽略"

### 文档区
- 表格：斑马纹、hover 高亮
- 代码块/流程图：灰底圆角框，等宽字体
- `[待确认]` 标签：橙色背景小标签（.tag-pending）
- `[设计建议采纳]` 标签：绿色背景小标签（.tag-added）
- 高亮块（.highlight-box）：#f0fdf4 绿色底色、#bbf7d0 边框
- 章节标题：蓝色下划线

### 文件命名
- `交互文档_[功能名称].html`（单文件，含反思+文档）

### 保存位置
保存到用户的工作目录，使用 `computer://` 链接提供访问。

## 内容精简原则

- **去掉**：数据字段定义、接口文档、技术实现细节、排期、埋点 keyName 详情
- **保留**：用户看到什么（UI 结构）、怎么交互（动作→反馈）、有几种状态（状态机）、文案是什么
- **精简表达**：表格优于段落，流程图优于文字描述，能用一行说清的不用三行
- **状态机必须完整**：每个可交互元素的所有状态都要覆盖
- **边界情况必须标注**：异常流、空状态、错误状态

---

## 分析原则

- **不遗漏**：PRD/原型图中每个可见元素都要覆盖
- **不臆断**：信息不足标注 `[待确认]` 而非猜测
- **可执行**：交互描述要足够明确，设计师能直接画稿
- **有层次**：从整体架构到具体页面，从主流程到异常分支
- **重状态**：每个组件的完整状态机是交互文档的核心价值

---

## 反思建议撰写原则

每条建议需满足：
1. **针对设计师可操作**：建议内容是设计师在交互稿中可以体现的，而非产品策略层面
2. **具体明确**：说清楚"在哪里"、"改什么"、"改成什么样"
3. **有理有据**：基于用户体验原则或可预见的使用场景

常见建议方向：
- 信息层级 / 密度调整
- 状态反馈不足（成功/失败/空态）
- 操作打断 / 流程中断的安抚
- 曝光频次 / 疲劳度控制
- 信息透明度（隐藏条件的前置展示）
