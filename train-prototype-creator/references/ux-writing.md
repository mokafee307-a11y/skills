# UX Writing Rules（预览页面文案）

本文件用于约束 **HTML 预览页面中用户会看到的文案**。它不替代 spec、IA 或交互说明，而是防止需求语言、系统语言、实现逻辑直接出现在用户界面中。

## 1. 核心原则

预览页面文案必须优先表达：

1. 用户收益：这个功能对用户有什么帮助。
2. 用户结果：用户完成后得到什么。
3. 用户场景：用户什么时候会需要它。
4. 用户行动：下一步可以做什么。
5. 屏幕效率：短文案优先，避免让 badge、button、chip、短说明因为措辞冗长而无谓换行。

预览页面文案避免表达：

1. 系统能力：支持、提供、可实现、可配置。
2. 实现方式：按日期分类、按状态筛选、数据同步、接口返回。
3. 产品设计说明：入口位置、交互路径、组件名称、浮层类型。
4. 内部逻辑：M-LIST、Half Screen View、lock、fallback、SVG、token。

一句话判断：

> 用户看到后应该立刻知道“这对我有什么用”，而不是知道“系统是怎么做的”。

## 1.1 业务语义边界

生成文案前必须先确认本轮需求的业务语义，不要让模板示例、上一轮项目或默认数据里的旧场景混入新页面。

执行方式：

- 先写出本轮核心目标，例如「积分抵票价感知」「收藏车次复用」「通勤快捷下单」。
- 再写出本轮明确不涉及的语义，例如「不涉及通勤」「不涉及收藏」「不涉及乘车人偏好」。
- 凡是不属于本轮目标的旧语义词和旧槽位，都不得出现在用户可见页面。
- 若模板默认文案与本轮目标不一致，必须重写或隐藏，而不是只在旁边追加新文案。

常见旧语义排除示例：

| 本轮需求 | 不应继承的旧语义 | 更合适的方向 |
|---|---|---|
| 积分抵票价 | 通勤、常买车次、快捷预订、乘车人偏好 | 可抵金额、抵后价、积分余额、金额明细 |
| 低价提醒 | 常用乘客、坐席偏好、直接占座 | 降价提醒、目标价、提醒结果 |
| 保险加购 | 收藏车次、开售查询、通勤复用 | 保障范围、价格、是否加购 |
| 选座优化 | 积分账户、收藏日期、快捷下单 | 座位偏好、车厢位置、上下车便利 |

## 2. 需求语言 → 用户语言

生成预览文案前，先把用户需求翻译成用户语言。

| 需求 / 系统语言 | 用户语言方向 |
|---|---|
| 支持收藏车次 | 常用车次快速查看 / 下次购票更方便 |
| 支持按收藏日期筛选 | 找回之前关注的车次 / 快速回看收藏方案 |
| 右下角固定入口 | 我的收藏 / 常用车次 / 已收藏 |
| Half Screen View 浮层 | 收藏车次 / 我的常用车次 |
| 支持低价监控 | 降价及时提醒 / 不错过好价 |
| 支持放票预测 | 不错过放票时机 |
| 支持智能推荐 | 更快找到合适方案 |
| 支持跨端同步 | 多设备都能查看 |
| 支持管理乘客 | 常用乘客更快填写 |
| 支持筛选排序 | 更快找到想要的车次 |
| 支持异常提醒 | 行程变化及时知道 |

## 3. 推荐文案结构

### 功能入口

优先短、明确、像 App 文案：

- 我的收藏
- 常用车次
- 已收藏
- 降价提醒
- 放票提醒
- 常用乘客

### 功能说明

用一句用户价值解释，不解释实现：

- 快速找到之前关注的车次
- 下次购票更方便
- 降价时及时提醒你
- 不错过合适的放票时机
- 常用信息自动带出，填写更省心

### 空状态

表达下一步行动：

- 还没有收藏车次
- 收藏常坐车次，下次查找更方便
- 看到合适车次，可以点星标收藏

### 状态反馈

少解释，多反馈：

- 已收藏
- 已取消收藏
- 已为你保存
- 可在“我的收藏”中查看

### 文案长度与换行

- 入口、badge、tab、button、chip 默认先争取单行展示。
- 如果少 4–6 个字不会损失用户收益，优先改短，不要用换行硬撑。
- 当文案已经短，但仍发生无谓换行时，优先调整容器宽度、间距和信息分层，而不是继续堆字。
- 只有长说明、空状态正文、票面必要信息才默认允许自然换行。

## 4. 禁止直接出现在 Preview 页面中的词

以下词可以出现在 spec / debug / changelog，但默认禁止出现在面向用户的预览页面中。若必须出现，需要改写为用户语言。

- 支持
- 筛选机制
- 排序逻辑
- 数据结构
- 分类管理
- 按收藏日期
- 按日期筛选
- 本地状态
- 跨端同步
- 登录同步
- Half Screen View
- 固定入口
- 组件
- 模板
- 页面基座
- data-uikit
- M-HOME / M-LIST / M-FILL / M-OCCUPY
- SVG / token / lock / fallback / manifest

注意：不是所有“筛选”都绝对不能出现。若是用户熟悉的控件名，如“筛选”，可以保留；但不要写成“支持按日期筛选收藏车次”这类解释系统能力的句子。

## 5. 文案自检

交付前对所有用户可见文案做一次检查：

1. 这句话是不是在解释功能怎么实现？如果是，重写。
2. 这句话是不是把 PRD 原文放进了页面？如果是，重写。
3. 用户不理解产品背景时，是否仍能看懂？如果不能，重写。
4. 是否出现具体测试日期、假数据日期、内部状态？如果不是票面/行程必要信息，删除或泛化。
5. 是否能压缩到 12 字以内的标题 + 20 字以内说明？优先压缩。
6. 按钮、badge、chip、路线摘要这类短文本，是否还有通过缩短措辞就能消掉的换行？如果有，继续改。
7. 这句话是否属于本轮业务目标？如果只是模板默认语义或上一轮场景残留，删除或替换。

## 6. 示例修正

### 收藏车次

错误：

> 收藏高频车次，临近出发按日期回看  
> 车次卡右上角可轻量收藏；右下固定入口随时查看已收藏车次，并按 2026-06-03、2026-06-02 等收藏日期筛选。

推荐：

> 我的常用车次  
> 快速找到之前关注的车次，下次购票更方便。

也可根据场景使用：

> 收藏车次  
> 看到合适车次，先收藏起来慢慢比较。

> 常坐车次  
> 经常乘坐的车次，一键查看。

### 低价提醒

错误：

> 支持订阅价格变化，并在价格低于阈值后推送通知。

推荐：

> 降价及时提醒  
> 价格合适时第一时间通知你。

### 放票预测

错误：

> 根据历史放票数据预测未来余票概率。

推荐：

> 不错过放票时机  
> 有票机会出现时及时提醒你。
