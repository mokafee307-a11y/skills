# Copy Review Checklist（用户可见文案审查）

每次生成或修改 HTML 预览后，必须对用户可见文案执行本检查。检查对象包括：标题、副标题、banner、toast、空状态、按钮、卡片说明、浮层标题与说明。

## 1. Preview Copy 规则

预览页面只能出现用户能理解、用户需要看到的产品文案。以下内容只能进入 `prototypes/*-spec.md`、HTML 注释、debug 文件或执行记录，不得直接显示在预览页面：

- 页面目标、成功标准、IA、设计说明。
- 组件名、模板名、页面模式 ID。
- SVG 复用记录、token、lock、fallback、manifest。
- 需求拆解、实现方式、内部数据字段。
- 测试日期、示例日期，除非它们本身是行程/票面信息。

## 2. 自动打回条件

用户可见文案出现以下问题时，必须重写：

1. 直接使用“支持 + 功能”的句式。
2. 用“按 xx 分类 / 按 xx 筛选 / 基于 xx 数据”等方式解释实现。
3. 暴露组件位置或交互实现，如“右下固定入口”“右上角按钮”“Half Screen View”。
4. 出现用户不需要理解的内部词，如 `M-LIST`、`data-uikit`、`SVG`、`token`。
5. 文案超过用户扫读负担：标题过长、副标题一口气解释多个能力。
6. 用户价值不清楚，只描述系统能做什么。
7. 按钮、badge、chip、tab、短标签、路线摘要等短文本在 375px 容器里出现明显可避免换行。
8. 出现与本轮需求无关的旧业务语义，如非通勤需求里出现「通勤 / 常买车次 / 快捷预订」，非收藏需求里出现「保存为常用车次 / 已收藏」。
9. 为了表达新增利益点而替换原有基础能力文案，如把「有票车次充足」「优先查看有票方案」改成优惠、积分或权益文案。新增利益点应使用 tag、提示、明细或承接页表达。

## 3. 重写方法

按以下顺序重写：

```text
原始需求
→ 本轮非目标 / 禁用旧语义
→ 用户场景
→ 用户收益
→ 短标题
→ 一句话说明
```

示例：

```text
需求：收藏车次，并按收藏日期查看
用户场景：用户想稍后比较或下次复用常坐车次
用户收益：快速找回之前关注的车次
标题：我的常用车次
说明：快速找到之前关注的车次，下次购票更方便
```

## 4. 长度建议

- 入口 / 按钮：2–6 个字。
- 卡片标题：4–10 个字。
- banner 标题：6–14 个字。
- 说明文案：12–24 个字。
- 空状态说明：12–28 个字。
- badge / chip / tab：2–6 个字优先。

超过长度不是绝对错误，但必须有明确价值，且不能像需求说明。
若文案本身已经不长，但仍然换行，先修布局和容器占比，不要直接接受换行结果。

## 5. 最终检查问题

交付前问自己：

- 用户会不会觉得这是 App 里该出现的话？
- 用户是否能 1 秒内看懂这句话的收益？
- 这句话删掉实现细节后，是否更像产品文案？
- 这句话是否可以更短、更具体、更像用户语言？
- 这句话是否来自当前需求，而不是模板示例或上一轮项目？
- 375px 容器里，是否还有通过压缩文案、调整容器、`nowrap` 或截断就能消掉的换行？

若答案不理想，重写后再交付。
