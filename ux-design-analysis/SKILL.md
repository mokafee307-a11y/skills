# UX Design Analysis V1.11

## 版本管理

- 版本号写在标题末尾（如 `V1.02`）
- 每次修改 +0.01；重大结构升级跳到下一个 0.1
- 修改后同步更新 description 前缀的版本号，并保存到 Cowork 注册

---

## 触发条件

用户发送产品 PRD 文档（PDF/图片/文字）并要求拆分为设计师需要的交互文档时使用此技能。

---

## 核心约束

1. **零杜撰，缺失标 [待确认]**：所有内容必须来自 PRD / 原型图 / 用户回答。信息不足时标 `[待确认]`，绝不虚构。
2. **面向设计师，去技术化**：去掉数据字段、接口、排期、埋点 keyName 等开发实现细节，专注「用户看到什么、怎么交互、有哪些状态」。
3. **反思先行**：先输出 3 个反思问题 + 建议，让用户确认后再生成文档。
4. **最多问 3 个问题**：信息不足时只挑最关键的 3 个让用户回答，其余写入 01 的待确认清单。
5. **章节默认收起**：01–05 五个章节均用「编号 + 标题 + 1-4 条核心信息卡片」展示，详情默认收起。
6. **输出即打开 Finder**：HTML 落盘后**必须**自动 `open -R`，不能仅回路径。

---

## 处理流程

### 第一步：读取输入

- **PDF 文件**：先 `pdftotext` 取文字，再 `pdftoppm` 转图片逐页查看原型图。**两步都做**，不能漏看视觉稿。
- **图片**：观察每个 UI 元素、状态、文案。
- **文字描述**：提取业务逻辑、交互规则、边界条件。

### 第二步：识别信息缺口 → 提 3 个最关键问题

只问会**阻塞设计**的问题（如未定义的核心交互、不明确的状态切换条件、关键文案缺失）。其他全部记入 01 待确认清单。

### 第三步：生成反思确认区 + 文档区一体化 HTML

**复用 `references/scaffold.html`**：
1. 读取该文件，把它的 **CSS 样式（`<style>` 块）和 JS 逻辑（`<script>` 块）整段复制**到输出文件——它是样式与功能的单一真相源，每次重新发明会漂移。
2. 用实际内容**替换**示例的 `<title>`、反思区 1 个问题、文档区 1 个章节为这份 PRD 的 3 个反思问题 + 5 个章节内容。
3. 在 `applyAdoption()` 函数体内根据本次 PRD 的反思建议写采纳后的 DOM 修改逻辑（修改对应单元格、ASCII 框图、状态机表）。

> 注：scaffold.html 不使用 `{{占位符}}` 模板语法，它本身就是一个可直接打开的最小可运行示例。复用方式是「读它取样式与 JS，扩展内容」，不是「字符串替换」。

### 第四步：保存到 PRD 同级的 `需求分析文档` 目录

文件命名 `交互文档_[功能名称].html`。

### 第五步：自动打开 Finder（必做）

```bash
open -R "/绝对路径/交互文档_{功能名称}.html"
```

回复中告知用户已在 Finder 中定位，并附 `computer://` 链接。**不要**自动打开浏览器。

### 第六步：输出前对照 `references/quality-checklist.md` 自检

逐项过 6 个维度（反思同步 / 摘要质量 / 章节身份 / 状态机完整 / 待确认清单 / 视觉一致性）。不通过的项回去修。

## 反思确认区（scaffold 已含 DOM 骨架，按需扩展内容）

### 三个固定问题

1. **这个需求的业务目标是什么？** — 分析需求本质
2. **目前这份PRD的做法，是否能完成这个目标？** — 评估方案可行性与隐忧
3. **这份PRD的做法是否会损害用户体验？** — 识别 UX 风险

### 建议撰写原则

**篇幅（必守）**：每个问题的 `.q-body` 与 `.suggestion-text` 各 **1–2 句话**，只留结论 + 最关键依据；细节放进生成后的 01/03，不在反思区展开。

每条建议还必须满足：
- **设计师可操作**：在交互稿中能体现（不是产品策略层面）
- **具体明确**：说清「在哪里 / 改什么 / 改成什么样」

**反例**（不要这样写）：
> 建议：优化用户体验，让操作更顺畅。

**正例**：
> 建议：模块顶部加「已根据习惯预选」+「修改」，自动设置可纠偏。

常见建议方向：信息层级 / 状态反馈 / 流程打断的安抚 / 曝光疲劳度 / 隐藏条件前置展示。

### 模块布局

- 3 个 `.section`（每个含问题卡片 + 建议卡片 + 同意/忽略 双按钮 + **状态徽章**）
- 1 个「其他建议」textarea
- 顶部 `.page-topbar`：左侧「需求反思确认 + 功能名」，右侧「全部忽略并生成」+「确认并生成」（默认 disabled，至少选过 1 条建议才可点）

**反思建议选择态（必做）**：点击「同意并修改」或「忽略」后须即时反馈——按钮高亮 + 文案变为「✓ 已同意」/「已忽略」；建议卡片右侧显示绿色 **已采纳** 或灰色 **已忽略** 徽章；卡片背景/边框随状态变化（`.is-agreed` / `.is-ignored`）。再次点击同一按钮可取消选择。

**建议卡片配色（必守）**：默认使用蓝色主题——背景 `#eff6ff`、边框 `#bfdbfe`、标签 `#dbeafe` / `#1e40af`、正文 `#1e3a8a`（与页面主色 `#2563eb` 一致）；勿用黄色警示色。

### 反思 → 文档同步规则

- **同意并修改** → 在 01 `#reflect-summary` 插入绿色 `[已采纳]` 高亮，**并同步修改** 03 的 ASCII 框图、交互动作表，04 的状态机表
- **忽略** → 按原 PRD 内容生成
- **其他建议**（textarea） → 文档末尾插入 `[补充建议]` 高亮，并修改涉及章节

## 文档结构

文档固定 **01–04**；仅当 PRD 存在边界场景（异常流、空态、错误态、特殊兜底）时增加 **05 边界场景**，否则**整章省略**，不写「本期暂无」。

### 章节展示形式（必守）

每个章节默认收起时只显示：
- **编号 + 标题**（如「01 背景与目标」，编号与标题平排）
- 下方 **1–4 条核心信息卡片**（每条含「序号 + sc-title + sc-desc」）
- 卡片间用分割线（`#eef0f3`），无背景色块
- sc-desc 中关键信息用 `<em class="kw">关键词</em>` 包裹（自动加 【】、文字色 `#0F4999` 加粗、无背景）

**展开后**：摘要卡片**自动隐藏**（CSS：`.chapter[open] > summary .chapter-summary-cards { display: none }`），只保留章节编号 + 标题 + 正文，避免与 h3 重复占位。这条样式已写在 scaffold 里。

### 标号统一规则（必守）

**所有章节内的细分标号一律使用大写字母 A / B / C / D**，不要使用 1 / 2 / 3 或 3.1 / 3.2。这条规则同时管：
- 摘要卡片的 `<span class="sc-tag">A</span>`（页面收起态可见的小方块标号）
- 展开正文里的 `<h3>A. 标题</h3>`（与摘要标号严格对齐）

即使一个章节只有 2 项，也按 A、B 排，不要因为只有 2 项就退化成 1、2。

**例**：
- 01 背景与目标 → A 一句话需求 / B 核心目标 / C 改动范围
- 02 用户场景 → A 首次进入 / B 日常买票 / C 异常态找替代 / D 购票后收藏
- 03 交互逻辑 → A 首页·自动设置 / B 首页·收藏车次卡片 / C 订单详情页·设为常买
- 04 状态说明 → A 日期组件 / B 下单按钮 / C 收藏按钮
- 05 边界场景 → A 接口异常 / B 操作中断 / C 网络兜底

### 摘要 ↔ 正文 融合规则（必守）

收起态的 sc-card 与展开态的 chapter-body 不应是「两块完全独立的内容」。融合方式：

- 摘要里的每张 sc-card 标题（如 `A. 一句话需求` / `B. 核心目标` / `C. 改动范围`）**直接作为展开后正文的 `<h3>` 小节标题**
- sc-desc 是该小节的精简引子，正文里直接接续详细内容（表格、ASCII 框图、blockquote 等），**不要把摘要文字再原样抄一遍**
- 如果某张卡片的内容确实只有一句话（如「一句话需求」blockquote），正文 h3 下面就只放那一句 + 必要的延伸说明，不强凑

**反例**：摘要写"核心目标：从 4700 → 30000"，展开正文又有一段"核心目标：…设置量从 4700 提升至 30000…" → 重复  
**正例**：展开后 `<h3>B. 核心目标</h3>` + 一段简洁延伸（"该提升靠首页自动设置 + 订单详情页一键收藏两路径达成"）+ 设计目标表

**例外**：**02 用户场景** 不按 ABCD 拆成多张子表 + h3 小节，而是**合并为单张表格**，表头加「场景」列写 ABCD 标号 + 场景名。理由：各场景字段同构（都是谁/何时/做什么/期望结果），合并更便于横向对比，分多张反而增加阅读成本。

### 各章核心卡片建议

| 章节 | 卡片数 | 卡片方向举例 |
|------|--------|-------------|
| **01 背景与目标** | 2-3 | 一句话需求 / 核心目标 / 改动范围 |
| **02 用户场景** | 3-4 | 各类典型场景（首次进入 / 日常使用 / 异常处理 / 后续动作） |
| **03 交互逻辑** | 2-4 | 各页面或模块的核心改动 |
| **04 状态说明** | 2-4 | 各核心组件的状态数 |
| **05 边界场景** | 2-4 | 各类边界（接口异常 / 操作中断 / 网络兜底） |

### 各章详情内容（展开后）

| 章节 | 写什么 | 不写什么 |
|------|--------|----------|
| **01 背景与目标** | 一句话需求（关键信息用【】 标记，如「针对【通勤购票用户】，通过【自动设置】，提升【设置量】」）/ 项目背景 / 设计目标表 / 观测指标表 / 需求反思（嵌套 `<details open>` **默认展开**）/ 涉及范围（嵌套 `<details open>` **默认展开**）/ 待确认清单 | 具体按钮点击、状态切换 |
| **02 用户场景** | **单张表格汇总所有场景**，表头依次为 `场景 \| 谁 \| 何时 \| 做什么 \| 期望结果`。「场景」列填 `A. 首次进入` / `B. 日常买票` 等 ABCD 标号 + 场景名（与摘要 sc-tag 严格对齐）。全程**用户视角**，期望结果写用户感知收益 | 业务目标、交叉率、AB 实验、引导转化（→ 01）；控件规则（→ 03）。**注意**：02 是融合规则的例外——不要按 ABCD 拆成多张子表 + h3 小节，因为各场景字段同构（都是谁/何时/做什么/期望结果），合并为一张表更便于横向对比 |
| **03 交互逻辑** | 按页面/模块分子节，每节含：改动点表 / 内容结构（ASCII 框图）/ 交互动作表（触发→反馈→反馈控件）/ 文案清单 / 流程图（如适用） | 完整状态枚举（→ 04） |
| **04 状态说明** | 按组件分子节，每节一张表 `状态 \| 展示条件 \| 视觉/文案 \| 可执行操作`，每个组件覆盖默认/进行中/成功/失败/空态 | 业务流程叙述（→ 03） |
| **05 边界场景** | 表格 `场景 \| 处理方式 \| 备注`，仅当 PRD 有边界规则时输出 | 主流程（→ 03）；PRD 缺口（→ 01 待确认） |

### 02「用户视角 vs 产品视角」对照

写 02 时，每行都问自己：「这是用户感受到的，还是公司想要的？」

| 类型 | 写到 02 ✅ | 不写到 02 ❌ |
|------|-----------|-------------|
| 谁 | "刚买到票、还需安排住宿的用户" | "命中实验组的用户" |
| 期望结果 | "信息清楚、不再用其他 App 查酒店" | "拉动酒店交叉率 +5%" |

### 待确认清单优先级口径

| 优先级 | 标准 | 例 |
|--------|------|-----|
| **高** | 不补就画不了稿（核心交互未定义） | "异常态跳转的具体动画形式" |
| **中** | 影响视觉/文案细节 | "引导弹窗动画时长" |
| **低** | 埋点、数据字段、排期 | "埋点设计" |

**用户已在反思 3 问中回答过的内容，从清单中移除**，直接写入正文，不再标 `[待确认]`。

### ASCII 框图最低标准

- 用 `┌─┐│└─┘` 等 box-drawing 字符；**移动端宽度约 40-46 字符**（手机屏幕宽度感）
- 每行后面用 `← 注释` 解释该区块作用
- 框图须**反映采纳后的状态**（如建议加了"调整 icon"，框图就要画出来）

## HTML 与样式

**直接读 `references/scaffold.html`，把它的 CSS（`<style>` 块）和 JS（`<script>` 块）整段复制到输出文件**——它是单一真相源。然后：

- 把示例 `<title>` 与两处副标题改为实际功能名
- 反思区把 scaffold 的 1 个示例问题扩展为 3 个实际问题（保持 `data-sid="q1/q2/q3"`）
- 文档区把 scaffold 的 1 个示例章节扩展为 5 个实际章节（编号 01–04，必要时加 05）
- 在 `applyAdoption()` 函数体内写本次 PRD 的反思采纳同步逻辑
- 替换保存文件名（`交互文档_示例功能名.html` → 实际名）

scaffold 已含的能力（不要重新发明）：
- 反思区按钮联动、确认/全部忽略 → 切到文档区
- 文档区：编辑模式、撤销重做（600ms 快照）、保存（File System Access API + 降级下载）、导出 PDF（窗口打印）、导出 Markdown（DOM 转换）、展开全部/收起全部、返回反思区
- 打印时隐藏顶部栏与反思区

### 关键样式约束（已写在 scaffold，仅作记录）

- **页面宽度**：反思区 max-width 960px / 文档区 max-width 1120px
- 强调色（`.kw`）：`#0F4999` 加粗，**无背景色**，自动加【】
- 卡片分割线：`#eef0f3`，无背景色块
- 标签：`[待确认]` 橙色 / `[已采纳]` `[补充建议]` 绿色
- 字号基础值 16px，章节标题 20px，章节卡片 sc-title 16px / sc-desc 15px
- 章节展开后 sc-card 自动隐藏：`.chapter[open] > summary .chapter-summary-cards { display: none }`
- **章节展开正文留白**：`.chapter-body { padding: 20px 32px 32px }`，表格 / pre / blockquote 不得贴卡片左右边缘

## 反馈控件选择规则

03 交互动作表中「反馈控件」列依据：

| 控件 | 适用场景 |
|------|---------|
| **内联更新** | 局部状态变化（按钮文案变、计数+1） |
| **Toast** | 操作结果轻量反馈（自动消失，不打断） |
| **浮层（Popup/Sheet）** | 补充信息或轻量操作集合 |
| **弹窗（Dialog/Modal）** | 需明确决策或确认（强制中断） |
| **页面跳转** | 信息量大或需独立上下文 |

**判断优先级**：能用内联就不用 Toast → 能用 Toast 就不用浮层 → 能用浮层就不用弹窗 → 能用弹窗就不用跳转。

**例外**：不可逆操作（删除、支付、退订）必须用弹窗二次确认。
