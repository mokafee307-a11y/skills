# 输出前自检清单

生成 HTML 之前，对照下面 6 项自检。每项不通过都要回去修正，不要心存侥幸提交。

---

## 1. 反思建议 → 文档同步检查

每条被采纳的反思建议，是否在以下位置都体现了？

- [ ] 01 章 `#reflect-summary` 内有对应的 `[已采纳]` 高亮块
- [ ] 03 交互逻辑：相关页面的「内容结构 ASCII 框图」「交互动作表」「文案清单」已修改
- [ ] 04 状态说明：相关组件的状态机表格已更新

**反例**（错误）：只在 reflect-summary 加高亮块，03/04 内容仍按原 PRD 描述 → 用户看到"采纳了"但文档没变。

---

## 2. 章节卡片摘要质量

01–05 每个章节的 1–4 条摘要卡片，是否符合：

- [ ] 每条 sc-title 是名词性短语（如「核心目标」「首次进入」），不是动词句
- [ ] 每条 sc-desc 一句话，10–25 字，含至少 1 个 `<em class="kw">关键词</em>`
- [ ] kw 内容是「数字、用户身份、关键功能名」，不是泛词
- [ ] 数量符合：01（2-3）、02（3-4）、03（按页面数 2-4）、04（按组件数 2-4）、05（边界类型 2-4）
- [ ] **sc-tag 与展开正文 h3 标号一律 A / B / C / D**，不混用 1 / 2 / 3
- [ ] **摘要标题已作为 h3 融合进展开正文**（不是「摘要 + 正文重复一遍」两块独立内容）
- [ ] **01 章内的「需求反思摘要」「涉及范围」默认展开**（`<details open>`），不收起

**反例**：`<em class="kw">优化用户体验</em>` ❌ 太抽象  
**正例**：`<em class="kw">4700 → 30000</em>`、`<em class="kw">售罄/已发车/停售</em>` ✅

---

## 3. 章节身份分工

- [ ] **02 用户场景**没出现「业务目标、交叉率、AB 实验、引导转化、产品策略」等产品视角词
- [ ] **02** 的「期望结果」列写的是用户感知（清楚、顺畅、不迷路），不是产品 KPI
- [ ] **03** 不写完整状态枚举（状态留给 04）
- [ ] **04** 不写业务流程叙述（流程留给 03）
- [ ] **05** 不写主流程（主流程在 03）

---

## 4. 状态机完整性（04）

每个可交互组件的状态机表格，是否覆盖了：

- [ ] 默认态
- [ ] 进行中 / loading（如适用）
- [ ] 成功态
- [ ] 失败态
- [ ] 空态 / 数据为空（如适用）
- [ ] 各种业务条件分支（如不可点的多种原因）

**反例**：只写「正常 / 异常」两态 → 设计师不知道异常下视觉、文案、操作如何。

---

## 5. 待确认清单（01）

- [ ] 用户在反思 3 个问题中**已回答的内容**，从待确认清单移除（不要再标 `[待确认]`）
- [ ] 清单条目按优先级排序：
  - **高**：阻塞设计、不补就画不了稿（如核心交互未定义）
  - **中**：影响视觉/文案细节（如动画形式、特定文案）
  - **低**：埋点、数据字段、上线时间
- [ ] 不强行创建占位条目（如"无"、"暂无"），少于 3 项就只列实际项

---

## 6. 视觉规范一致性

- [ ] 所有「关键信息」用 `<em class="kw">…</em>`，**无背景色、文字色 #0F4999、自动加【】**
- [ ] 所有 ASCII 框图 / 流程图放在 `<pre>` 中，等宽字体可读
- [ ] 卡片之间用分割线 `#eef0f3`（无背景色块）
- [ ] tag 小标签：`[待确认]` 橙色 / `[设计建议采纳]` 或 `[补充建议]` 绿色
- [ ] 文档末尾未夹带「仅画实验组」「对照组方案」等无效说明（这些是给 Claude 的内部约束，不要露给用户）
